# خطة بناء نظام طباعة احترافي داخل Electron

(Enterprise Printing System – Phased Plan)

## الهدف العام

بناء نظام طباعة موحّد داخل تطبيق Electron يدعم:

- قوالب Word قابلة للتحرير
- ترويسة وذيل ديناميكيين
- تصدير Word و PDF
- RTL عربي مضبوط
- إعدادات طباعة مستقلة
- محرك واحد يخدم كامل النظام

---

## المرحلة 1: التأسيس والهيكلية (Foundation)

**المطلوب:**
- إنشاء هيكل مجلدات خاص بالطباعة داخل main process
- فصل منطق الطباعة عن واجهة المستخدم
- اعتماد محرك طباعة واحد لكل المستندات

**المخرجات:**
- Folder structure جاهز
- ملف PrintEngine أساسي
- نقطة دخول موحّدة للطباعة

**معايير القبول:**
- لا يوجد كود طباعة داخل renderer
- كل الطباعة تتم عبر main process فقط

---

## المرحلة 2: نظام القوالب (Template System – DOCX)

**المطلوب:**
- اعتماد DOCX كقالب أساسي
- دعم متغيرات ديناميكية داخل القالب
- عدم ربط القالب بالكود

**أمثلة متغيرات:**
- `{{tenant_name}}`
- `{{owner_name}}`
- `{{contract_amount}}`
- `{{start_date}}`

**المخرجات:**
- مجلد `templates`
- قالب واحد تجريبي يعمل (عقد إيجار)

**معايير القبول:**
- يمكن تعديل القالب بدون إعادة بناء التطبيق
- دعم RTL بشكل صحيح

---

## المرحلة 3: الترويسة والذيل (Header / Footer Engine)

**المطلوب:**
- ترويسة قابلة للتعديل (شعار – اسم الشركة – وصف)
- ذيل ذكي (رقم الصفحة – التاريخ – اسم المستخدم)
- إمكانية تشغيل/إيقاف الترويسة والذيل من الإعدادات

**المخرجات:**
- Header template
- Footer template
- Injector يدمجهم مع القالب

**معايير القبول:**
- لا يوجد Header ثابت داخل القالب
- الترويسة والذيل ديناميكيين

---

## المرحلة 4: إعدادات الطباعة (Print Settings)

**المطلوب:**
- ملف إعدادات مستقل (JSON)
- دعم:
  - حجم الصفحة
  - الهوامش
  - الاتجاه
  - الخط
  - RTL
  - إظهار/إخفاء الترويسة والذيل

**المخرجات:**
- `print.settings.json`
- Loader + Saver للإعدادات

**معايير القبول:**
- الإعدادات تطبق على جميع المستندات
- لا Hard-coded values

---

## المرحلة 5: محرك التوليد (Generation Engine)

**المطلوب:**
- توليد DOCX نهائي من:
  - قالب
  - بيانات
  - ترويسة
  - ذيل
- حفظ نسخة مؤقتة آمنة

**API مقترح:**
```ts
generateDocument({
  template,
  data,
  outputType,
})
```

**معايير القبول:**
- محرك واحد لكل أنواع المستندات
- قابل للتوسعة لاحقًا

---

## المرحلة 6: تصدير PDF (PDF Export)

**المطلوب:**
- تحويل DOCX إلى PDF بدون كسر التنسيق
- دعم الخط العربي و RTL
- نفس الناتج البصري للـ Word

**السيناريو المعتمد:**
- DOCX → Render → PDF

**معايير القبول:**
- PDF مطابق 100% للـ DOCX
- لا تشويه ولا قص صفحات

---

## المرحلة 7: المعاينة (Print Preview)

**المطلوب:**
- نافذة معاينة داخل Electron
- عرض PDF مؤقت
- أزرار:
  - طباعة
  - تصدير Word
  - تصدير PDF

**معايير القبول:**
- لا طباعة بدون معاينة
- المعاينة سريعة ومستقرة

---

## المرحلة 8: الطباعة المباشرة (Direct Printing)

**المطلوب:**
- الطباعة عبر Electron API
- دعم اختيار الطابعة
- دعم الخلفيات والهوامش

**معايير القبول:**
- لا طباعة صامتة افتراضيًا
- احترام إعدادات النظام

---

## المرحلة 9: الصلاحيات (Permissions)

**المطلوب:**
- ربط الطباعة بنظام الصلاحيات
- أدوار:
  - موظف: طباعة فقط
  - مشرف: تعديل القوالب
  - مدير: تعديل الترويسة
  - Admin: كل الصلاحيات

**معايير القبول:**
- لا تعديل بدون صلاحية
- فحص الصلاحيات قبل التنفيذ

---

## المرحلة 10: التوحيد والربط مع النظام

**المطلوب:**
- زر طباعة موحّد في جميع الشاشات
- إرسال:
  - نوع المستند
  - ID
  - البيانات

**معايير القبول:**
- لا منطق طباعة مكرر
- كل شيء يمر عبر PrintEngine

---

## المرحلة 11: الاختبار والجودة (QA)

**المطلوب:**
اختبار:
- Word
- PDF
- طابعة فعلية
- RTL
- أسماء طويلة
- أكثر من صفحة

**معايير القبول:**
- لا قص محتوى
- لا أخطاء تنسيق
- استقرار كامل

---

## النتيجة النهائية

- نظام طباعة احترافي
- قابل للتوسع
- سهل الصيانة
- منفصل عن منطق النظام
- مناسب للاستخدام التجاري طويل المدى

---

## ملاحظات إلزامية

- يمنع Hard-code لأي إعداد
- يمنع منطق طباعة داخل UI
- أي إضافة يجب أن تكون Extendable
- الالتزام بالـ RTL في جميع المراحل

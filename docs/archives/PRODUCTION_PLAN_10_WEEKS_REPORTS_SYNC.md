# خطة إنتاج خلال 10 أسابيع (Reports + Sync) — مع أحجام بيانات كبيرة

**الهدف:** إصدار Production خلال 10 أسابيع لمكتب واحد الآن، مع التركيز على:
1) **التقارير** (مالية/عقود/عقارات/مستأجرين/صيانة ودعم) بجودة عالية
2) **المزامنة**: **SQL Server Sync كجاهزية مستقبلية** فقط (قابل للتطوير لاحقاً)، مع تقديم **Backup/Restore** كحل حماية بيانات أساسي الآن.

**حجم البيانات المتوقع:**
- الأشخاص: > 200,000
- العقارات: ~ 100,000
- العقود: ~ 100,000

> ملاحظة حرجة: التخزين الحالي يعتمد على مفاتيح مثل `db_people`, `db_properties`, `db_contracts` كـ JSON arrays داخل KV (حتى في Desktop SQLite فهو KV فقط).
> هذا **لن يصمد** مع هذه الأحجام (زمن بدء طويل، استهلاك RAM ضخم، parsing JSON مكلف، تقارير بطيئة).
> لذلك الخطة تتضمن **ترحيل طبقة البيانات إلى جداول SQLite حقيقية** مع فهارس واستعلامات Paging.

---

## مبادئ تقنية ثابتة (Non‑negotiables)

1) **لا تحميل لكل البيانات للواجهة دفعة واحدة**
- أي صفحة/تقرير: تستخدم `paging` + `filters` + `server-side aggregation`.

2) **التقارير تُبنى على SQL** قدر الإمكان
- الحسابات والتجميعات في قاعدة البيانات أسرع وأكثر دقة من التجميع داخل React.

3) **عزل طبقة البيانات**
- Renderer لا يقرأ JSON ضخمة من storage.
- Renderer يطلب بيانات عبر IPC (استعلامات محددة) أو عبر واجهة `window.desktopDb`.

4) **Backup/Restore أولاً** (مكتب واحد)
- قبل أي مزامنة متعددة الأجهزة: حماية بياناتك من الفقد.

---

## ناتج نهائي للإصدار (Definition of Done)

- التقارير الخمسة (Finance/Contracts/Properties/Tenants/Maintenance) موجودة ضمن شاشة واحدة أو صفحات واضحة.
- كل تقرير:
  - فلاتر (فترة/حالة/موظف/عقار/مستأجر)
  - أداء جيد (لا تجميد UI)
  - تصدير Excel موثوق
  - نتائج متطابقة مع قواعد الحساب المعتمدة
- النسخ الاحتياطي والاستعادة:
  - Backup يدوي + مجدول
  - Restore مع فحص سلامة + تأكيدات
- (جاهزية مستقبلية) SQL Server Sync: تصميم قابل للتوسعة + سجل مزامنة + retry، **بدون إلزام تشغيله في الإصدار الحالي**.

## خارطة طريق 10 أسابيع (عمليّة)

### الأسبوع 1 — تحديد نطاق التقارير + تصميم البيانات
**مخرجات:** وثيقة متطلبات + قائمة تقارير MVP + تصميم DB
- حصر تقارير الـ MVP (حد أدنى):
  - **مالية:** تحصيلات/متأخرات/إيراد شهري/عمولات
  - **عقود:** سارية/منتهية/قريبة الانتهاء/تجديدات
  - **عقارات:** شاغرة/مؤجرة/جودة البيانات
  - **مستأجرين:** قائمة/مخاطر (سلوك دفع)/متأخرات حسب المستأجر
  - **صيانة:** تذاكر مفتوحة/مغلقة/حسب العقار/حسب الحالة
- تثبيت تعريفات الحساب (مثلاً: متأخر = تاريخ استحقاق < اليوم && غير مدفوع).
- نطاق التقارير المالية في هذا الإصدار: **(تحصيلات/متأخرات/إيرادات/عمولات) فقط** — بدون مصروفات.
- قرار معماري: **SQLite Domain Tables** على Desktop.

**قرار مهم:**
- الإبقاء على KV للـ Settings/Lookups فقط.
- نقل الكيانات الضخمة إلى جداول حقيقية: People/Properties/Contracts/Installments/Commissions/Maintenance.

### الأسبوع 2 — بناء قاعدة بيانات Domain داخل SQLite (بدون كسر النظام)
**مخرجات:** DB schema + مهاجر (migration) + فهارس
- إنشاء جداول (مثال عالي المستوى):
  - `people(id, name, national_id, phone1, phone2, createdAt, updatedAt, ...)`
  - `properties(id, internal_code, owner_id, status, water_no, electric_no, area, ..., updatedAt)`
  - `contracts(id, property_id, tenant_id, start_date, end_date, status, value, ..., updatedAt)`
  - `installments(id, contract_id, due_date, paid_date, status, amount, remaining, ..., updatedAt)`
  - `maintenance_tickets(id, property_id, status, opened_at, closed_at, cost, ..., updatedAt)`
  - (اختياري) `commissions(...)`
- فهارس أساسية (Indexes):
  - people: `phone`, `national_id`, `name`
  - contracts: `tenant_id`, `property_id`, `status`, `end_date`
  - installments: `contract_id`, `due_date`, `status`
  - properties: `internal_code`, `owner_id`, `status`
- إدخال FTS5 للبحث النصي (إن أمكن): people.name, properties.internal_code.

**ملاحظة:** هذا الأسبوع يركز على DB + IPC فقط (لا UI).

### الأسبوع 3 — جسر IPC لاستعلامات Paging + تحويل صفحات رئيسية للقراءة الجزئية
**مخرجات:** APIs جاهزة + أول صفحة تستفيد من paging
- إضافة استدعاءات مثل:
  - `queryPeople({ q, page, pageSize, sort })`
  - `queryContracts({ status, dateRange, tenantId, page, pageSize })`
  - `queryProperties({ occupancy, code, ownerId, page, pageSize })`
- أول صفحة تتحول لعرض جزئي (مثلاً People):
  - بحث + Paging + Virtualized list

### الأسبوع 4 — التقارير المالية (أولوية 1)
**مخرجات:** لوحة تقارير مالية + تصدير Excel
تقارير مالية MVP:
- التحصيلات حسب الفترة (مدفوع/غير مدفوع/متأخر)
- متأخرات الكمبيالات (Top N + إجمالي)
- إيراد شهري (آخر 12 شهر)
- العمولات (حسب موظف/شهر)

**ملاحظة:** لا يتم تضمين المصروفات ضمن هذا الإصدار.

تقنياً:
- تنفيذ queries SQL للتجميعات.
- تصدير Excel:
  - عند أحجام كبيرة: التصدير يجب أن يكون **Chunked/Streaming** (يفضل في Electron main) لتجنب تجميد UI.

### الأسبوع 5 — تقارير العقود + المستأجرين
**مخرجات:** تقارير سريعة قابلة للفلترة والتصدير
- العقود السارية/قريبة الانتهاء/منتهية
- تجديدات خلال 30/60/90 يوم
- قائمة مستأجرين مع مؤشرات: عدد العقود، متأخرات، آخر دفع

### الأسبوع 6 — تقارير العقارات + الصيانة والدعم
**مخرجات:** تقارير تشغيلية
- عقارات شاغرة/مؤجرة + نسب إشغال
- جودة بيانات العقار (نواقص)
- صيانة: مفتوح/قيد التنفيذ/مغلق + متوسط زمن الإغلاق + تكاليف

### الأسبوع 7 — Backup/Restore (حقيقي لمكتب واحد)
**مخرجات:** حماية بيانات + سلامة
- Backup:
  - زر “نسخ احتياطي الآن” (يحفظ نسخة من sqlite + attachments إن وجدت)
  - جدولة يومية/أسبوعية (اختياري)
- Restore:
  - اختيار ملف backup
  - فحص سلامة + تأكيد + استعادة + إعادة تشغيل/Reload

### الأسبوع 8 — مزامنة (اختياري) + سجل مزامنة قوي
**مخرجات:** جاهزية للمستقبل (SQL Server Sync)
- بما أن الهدف الحالي "قابل للتطوير لاحقاً" فقط:
  - تنفيذ **تصميم** قابل للمزامنة بدون إجبار تشغيله:
    - Outbox/Change log أو أعمدة `updatedAt`/`deletedAt` على مستوى الصف
    - توحيد IDs + سياسة تضارب (Conflict Policy) واضحة (مثلاً Last-Write-Wins مع استثناءات حقول حساسة)
  - شاشة سجل مزامنة (Logs) + retry/backoff + مؤشرات تقدم.
- يبقى الاعتماد التشغيلي في الإصدار الحالي على: **Backup/Restore + خيار DB path (OneDrive) عند الحاجة**.

### الأسبوع 9 — تحسين أداء شامل + اختبارات صحة التقارير
**مخرجات:** ثقة بالأرقام
- اختبارات “صحة” لقواعد الحساب (أمثلة بيانات + نتائج متوقعة).
- Profiling للشاشات الثقيلة.
- إصلاحات UI (Virtualization, debounced search, caching query results).

### الأسبوع 10 — تجهيز الإصدار + تجارب تشغيل واقعية
**مخرجات:** Release جاهز
- سيناريو يوم عمل كامل:
  - إدخال بيانات → تحصيلات → تقارير → backup/restore
- إصلاحات أخيرة + توثيق مختصر للمستخدم.

## مخاطر رئيسية وكيف نعالجها

1) **تحميل JSON ضخم** (حاليًا)
- العلاج: نقل إلى SQLite domain tables + paging.

2) **تصدير Excel لأكثر من عشرات الآلاف من الصفوف**
- العلاج: تصدير من Electron main أو chunked + progress.

3) **تضارب مزامنة مستقبلية**
- العلاج: تصميم row-level updatedAt + policy واضحة.

## افتراضات مثبتة (حسب قراراتك)

- التقارير المالية: **تحصيلات/متأخرات/إيرادات/عمولات فقط**.
- SQL Server Sync: **جاهزية مستقبلية** (قابل للتطوير)، وليس شرطاً تشغيلياً للإصدار الحالي.
